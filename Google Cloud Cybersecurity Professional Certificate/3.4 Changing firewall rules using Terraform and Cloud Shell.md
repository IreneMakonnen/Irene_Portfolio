# Changing firewall rules using Terraform and Cloud Shell

Firewall rules play a crucial role in cloud network security because they control which traffic is allowed to enter and leave the cloud environment. The nature of cloud networking is ever changing and complex. Organizations need change, new cloud resources are added or removed, and new vulnerabilities are discovered. This means that we'll need to continually adjust and maintain the configuration of firewall rules to adapt with these changes.

Tracking and managing critical updates in the cloud environment can be challenging. Luckily, you can manage the configuration of cloud resources by leveraging **infrastructure as code (IaC)**. IaC is the provisioning and managing of infrastructure through using reusable scripts. It can be used to automate workflows such as updating firewall rules.

Terraform and Cloud Shell make it easy to manage and update firewall rules efficiently. By writing desired firewall rules in Terraform configuration files and executing Terraform commands in Cloud Shell, you can ensure that your network security policies remain consistent and version-controlled. This approach will help you maintain and track changes over time.

## Scenario

Cymbal Bank's new banking application is ready to be deployed. It needs to be hosted on a Virtual Private Cloud (VPC). Your team lead, Chloe, would like to define and provision the application's network infrastructure using Terraform. Terraform can be used to securely provision cloud infrastructure in a way that's repeatable and consistent. This way, you can easily and quickly make any required changes. You have been tasked with using Terraform to create a network and firewall rules.

## Cloning the Terraform repo

In this task, we'll clone the Terraform example repository using the Cloud Shell terminal. The Terraform example contains the configuration file, which we'll use to provision the firewall rules.

1. In the Google Cloud console, clicked the **Activate Cloud Shell** `>_`

2. Typed the following command into the Cloud Shell terminal:

```HashiCorp Configuration Language
cloudshell_open --repo_url "https://github.com/terraform-google-modules/docs-examples.git" --print_file "./motd" --dir "firewall_basic" --page "editor" --tutorial "./tutorial.md" --open_in_editor "main.tf" --force_new_clone
```

This command clones the Terraform example directory.

This command performs the following actions:

- Clones the `terraform-google-modules`.
- Prints the `motd` file name.
- Switches to the `firewall_basic` directory.
- Checks the cloned files, for example `tutorial.md`.
- Opens `main.tf` in Cloud Shell Editor.

Once the cloning is complete, we’ll be at the `~/cloudshell_open/docs-examples/firewall_basic` location in the terminal. The Cloud Shell prompt should display similar output to the following example:

```
student_01_c2e095df84e2@cloudshell:~/cloudshell_open/docs-examples/firewall_basic (qwiklabs-gcp-04-fde36f013e65)$
```

3. Typed the following command into the Cloud Shell terminal to list the contents of the directory:

```sh
ls
```

You should notice that several files in the directory have been downloaded: `backing_file.tf`, `main.tf`, `motd`, and `tutorial.md`.

4. Typed the following command into the Cloud Shell terminal to analyze the configuration of the firewall rule:

```sh
cat main.tf
```

The `main.tf` file is the configuration file that defines the resources that Terraform will create. Two resources will be created: a firewall rule `google_compute_firewall` named `test-firewall-${local.name_suffix}` with rules to allow ICMP and TCP traffic from ports 80, 8080, and 1000-2000 and a VPC network `google_compute_network` named `test-network-${local.name_suffix}`. The variable `${local.name_suffix}` is a local variable that automatically generates unique names for resources.

The ICMP and TCP protocols are being modified (allow/deny) on the firewall using the Terraform **main.tf** file in **Cloud Shell Editor**.

## Deploying the VPC network and firewall

In this task, we'll deploy a new VPC network and a new firewall rule. This task provides hands-on experience with building a VPC network and subnets.

1. Typed the following command into the Cloud Shell terminal.

```
export GOOGLE_CLOUD_PROJECT=Project ID
```

This command sets the project ID.

2. Typed the following command into the Cloud Shell terminal:

```HashiCorp Configuration Language
terraform init
```

This command initializes the Terraform script.

The output should return a message stating that the Terraform has been successfully initialized. Took a moment to examine the output. We'll notice that Terraform will create a new firewall and VPC network:

![Successful Terraform initialization](<attachments/3. Cloud Security Risks- Identify and Protect Against Threats/Successful Terraform initialization.png>)

3. Once the initialization is complete,typed the following command into the Cloud Shell terminal:

```HashiCorp Configuration Language
terraform apply
```

This command applies the changes and deploys the Terraform script.

_**Note:** If an **Authorize Cloud Shell** dialog box appears, click **Authorize** to grant permission to use your credentials for the gcloud command._

4. The command prompt will prompt to **Enter a value**. Typed "yes", and pressed **ENTER**.

This will start creating the VPC network and firewall rules.

Once it’s completed, the output should return the following message:

```
Apply complete! Resources: 3 added, 0 changed, 0 destroyed.
```

This means that the VPC and firewall have been successfully deployed.

## Verifying the deployment of the resources

In this task, we'll verify that the newly created VPC and firewall rules have been successfully deployed.

1. In the Google Cloud console, from the Navigation menu (☰), selected **VPC network > VPC networks**. The VPC networks page opens.
2. Notice two VPC networks, **default** and the newest one that was just created, **test-network**. Clicked **test-network** to access the VPC network details.
3. Clicked **Firewalls**. Used the expand arrow to expand **vpc-firewall-rules**. Under **Protocols and ports** and **Action**, notice the firewall rules are the same rules as defined in the configuration file: **Allow** and **tcp:80**, **1000-2000**, **8080 icmp**.

_**Note:** To ensure that resource names are unique, both the **test-network** and **test-firewall** names will be dynamically appended with a unique identifier. For example, **test-network-curly-penguin**. This unique identifier is generated automatically by the **${local.name_suffix}** local variable, which is defined in the configuration file. This helps prevent resource naming conflicts and ensures the proper organization of infrastructure components._
